Grille stylée sans grid, éléments générés par js

Mon objectif est de rendre une grille en utilisant des directives css standards (pas de grid), et d'utiliser Javascript pour la génération des cases.

Je voulais avoir une maîtrise exacte de la taille du plateau, et que le calcul de la taille des cases ne soit pas affecté par les marges et les bordures.

Pour ce dernier point, les éléments déterminants ont étés l'utilisation de la directive css box-sizing: border-box, ainsi que l'encapsulation de chaque ligne de la grille dans un div dédié. En effet sans cette encapsulation, les cases, chacune rendue par un div et par une directive css display: inline-block, s'alignaient correctement par lignes, mais il restait des marges verticales entre celles-ci. Je pense que ces marges étaient induites par le inline du inline-block. En encapsulant les cases d'une ligne dans un div, élément nativement block, je me suis affranchi de cet effet inline.

Le résultat est correct, et je peux jouer presque facilement sur le nombre de cases du plateau. Presque, car si côté Javascript ce nombre est simplement un argument de la fonction drawSquareGrid (il pourrait être paramétrable via une propriété data-nb-case-per-row de l'élément plateau, par exemple), il faut encore modifier les directives css height: 10%; et width: 10%; – 10% car 10 cases par côté codé en dur. Je soupçonne que l'utilisation de l'unité relative fr devrait m'aider. To be continued...

Grille paramétrable, sans grid, éléments générés par js

L'idée est de reprendre l'implémentation précédente, pour que le nombre de cases soit facilement paramétrable, avec uniquement l'argument nbCellPerRow de la fonction drawSquareGrid
→ pas de style height et width en dur dans le css (ou alors valeur calculée en fonction du nombre de cases par côté)
→ découpler au maximum le modèle (js) de la vue (html et css)

J'ai amélioré le découplage modèle - vue notamment pour l'application des bordures haute et gauche des cases les plus en haut, et les plus à gauche. Voir les ressources utilisées (lien ci-dessous) Mais je n'ai pu m'affranchir de positionner dans le code js les width et height des cellules. Impossible de le faire dans la css, car ce sont des valeurs calculées suivant le nombre de cellule par côté. L'utilisation de la fonction css counter permet de calculer ce nombre; j'espérais ensuite pouvoir l'utiliser avec la fonction calc pour déterminer la valeur... mais sans succés. On va considérer que le résultat est acceptable, cela reste un exercice, qui m'a permis de m'exercer encore un peu plus aux css !

Échapper les caractères HTML : oui mes lesquels ?

Très peu de caractères sont échappés dans le code html de cette page (par exemple é pourrait être échappé &eacute;), parce qu'elle déclare utiliser le jeu de caractères UTF-8 grâce à la balise <meta charset="utf-8"> dans le header.
Ainsi, pour peu que le code html soit encodé en utf-8 (mon IDE est configuré pour cela), et que le navigateur qui lit cette page connaît utf-8 (ce que l'on peut considérer comme étant le cas pour tous les navigateurs), les caractères seront rendus tels que je les ai écrits.

Pour être exhaustif, il faut s'assurer aussi que le serveur http ne surcharge pas la déclaration de la balise meta. Il peut le faire, en précisant le jeu de caractères dans une en-tête de réponse Content-Type: text/html; charset=XXX (XXX étant un code de jeu de caractères autre que utf-8, par ex. iso-8859-1) Ce serait fâcheux et... surprenant !
